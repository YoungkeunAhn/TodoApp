# ADR 0009 — 카테고리 수정·삭제(삭제는 할 일까지 CASCADE)

- 상태: **채택됨**
- 관련: [카테고리 기능](../category.md), [데이터 구조](../data-model.md), [0007 카테고리 이름 중복 방지](./0007-unique-category-name.md)

## 맥락 (Context)

- 그동안 카테고리는 **추가·조회**만 가능했고, 한번 만든 이름을 고치거나 지울 수 없었다.
- 오타·이름 변경·정리를 위해 **수정/삭제**가 필요했다.
- 삭제 시 핵심 질문: 그 카테고리를 쓰는 **할 일을 어떻게 할 것인가?**
  - `todos.category_id`는 `not null`이라 "카테고리 없음" 상태로 남길 수 없다.
  - 스키마(`schema.sql`)는 이미 `references categories(id)` **`ON DELETE CASCADE`** 로 선언돼 있다.

## 결정 (Decision)

- **수정**: 사이드바 카테고리 항목 hover 시 나오는 **✎** 버튼 → `prompt`로 새 이름 입력 →
  `categories.name` 업데이트.
  - 중복 검사는 추가와 동일하게 **자기 자신 제외** + 대소문자/공백 무시, DB UNIQUE(`23505`)도 안내로 변환([0007](./0007-unique-category-name.md)).
  - 수정 후 `loadTodos()`까지 다시 불러 **할 일 목록에 보이는 카테고리 이름**도 함께 갱신한다.
- **삭제**: **×** 버튼 → `confirm` → 행 삭제. 기존 스키마의 **`ON DELETE CASCADE`를 그대로 따른다.**
  - 즉 카테고리를 지우면 **그 카테고리의 할 일도 함께 삭제**된다(별도 차단/이전 없음).
  - 파괴적 동작이므로, 연결된 할 일이 있으면 확인창에 **"할 일 N개도 함께 삭제됩니다"** 로
    개수를 보여 주고 되묻는다(개수는 메모리 캐시 기준).
- **UI**: 액션 버튼은 todo의 삭제 버튼과 같은 **hover 노출** 패턴을 쓰고, 터치 기기
  (`@media (hover: none)`)에서는 항상 보이게 한다. 버튼 클릭은 `stopPropagation`으로
  모바일 사이드바가 닫히지 않게 한다.
- **`app.js` 구조 유지**: 기존 add/load 흐름과 동일하게 DB 변경 후 관련 목록을 다시 불러 그린다.

## 결과 (Consequences)

**좋은 점**
- 카테고리를 자유롭게 정리할 수 있고, 이름 변경이 할 일·통계에 일관되게 반영된다.
- 삭제 동작이 스키마(CASCADE)와 **일치**해, "지웠는데 할 일이 붕 뜨는" 모순이 없다.

**감수하는 점**
- 카테고리 삭제가 **할 일까지 지우는 파괴적 동작**이다. 확인창 경고로 완화하지만, 되돌리기(undo)는 없다.
  - 더 안전한 대안(삭제 차단=`RESTRICT`, 또는 "미분류"로 이전)은 스키마 변경이 필요해 이번 범위에서 제외했다.
- 함께 삭제될 할 일 **개수는 메모리 캐시 기준**이라, 다른 기기에서 방금 추가된 할 일은 셈에서 빠질 수 있다.
- 수정/삭제 UX는 `prompt`/`confirm` 기본 대화상자를 쓴다(정적 사이트 단순성 우선, [0002](./0002-vanilla-frontend.md)).
